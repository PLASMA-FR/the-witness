Overview

1. Executive Summary

The Witness is a TUI-based local proxy that lets users add AI endpoints, watch every request and response, verify each response using Gemma 4, block bad answers, repair prompts, retry, and only return approved responses.

“The Witness is not another chatbot.”
It is the verification layer between AI generation and real-world action.

Instead of trusting the first response from an upstream model, The Witness asks a local Gemma 4 judge whether the response is safe, useful, aligned with the prompt, and low-risk enough to pass through.

$ the-witness start

Dashboard → Endpoint Watchlist → Live Stream → Gemma Verdict → Prompt Repair → Audit Log
Why it matters

2. The Problem

AI tools are becoming agents. They write code, tutor students, summarize health information, support finance workflows, and automate work. But many tools trust the first model response immediately.

Incorrect code

A weak answer can create broken scripts, unsafe shell commands, or insecure defaults.

Bad tutoring

Students may see plausible but wrong explanations before anyone reviews them.

Overconfident advice

Health, finance, and legal topics require caution, uncertainty, and human review.

Hallucinated facts

Agents may invent APIs, citations, or operational details.

Untraceable decisions

Without logs, teams cannot reconstruct what was approved, rejected, or retried.

Hidden risk

The failure happens between the model and the user. The Witness puts a gate there.

Risk statement: The point is not to claim perfect safety. The point is to add a local, explainable verification gate before model output reaches real users.
Architecture

3. The Solution

The Witness sits between the AI app and the upstream model endpoint. The original application calls a localhost OpenAI-compatible proxy. The Witness forwards the request, captures the candidate response, and asks Gemma 4 for a structured verdict before releasing anything.

  • watches the endpoint
  • captures request/response pairs
  • sends candidate responses to Gemma 4
  • approves, rejects, repairs, retries, or pauses for human review
  • writes an audit trail

Usage flow

4. Core Workflow

DISAPPROVED

Block candidate response
Generate prompt repair
Retry upstream endpoint
⇄
APPROVED

Return final response
Mark request approved
Save audit log
  1. User adds endpoint.
  2. The Witness creates a local proxy URL.
  3. AI app sends request to local proxy.
  4. The Witness forwards request to upstream endpoint.
  5. Upstream returns candidate response.
  6. Gemma 4 judges response.
  7. If approved, response is returned.
  8. If disapproved, response is blocked.
  9. Prompt repair is generated.
  10. Request is retried.
  11. If risky, human review queue is used.
  12. Logs are saved.
Approval loop: response capture, Gemma verdict, disapproval, prompt repair, retry, approval, and audit.
Capabilities

5. Features

TUI dashboard

Live operational view of endpoint health, request stats, approvals, rejections, retry counts, and current judge configuration.

Why it matters: Judges and users can see the system working without opening a browser.

Endpoint watchlist

Add, edit, enable, disable, test, and duplicate watched endpoint configurations.

Why it matters: Every endpoint can have its own proxy URL, profile, retry limit, strictness, and fallback behavior.

OpenAI-compatible local proxy

Applications point at localhost while The Witness forwards to the real upstream endpoint.

Why it matters: Existing tools can adopt the verification layer with minimal configuration changes.

Request/response monitoring

Captures model name, prompt content, metadata, latency, and candidate response before release.

Why it matters: Turns invisible model traffic into an auditable timeline.

Gemma 4 judge

A local Gemma 4 model reviews candidate responses and returns structured safety and quality verdicts.

Why it matters: Keeps the trust layer local-first and inspectable.

Structured JSON verdicts

The judge must output a strict schema with verdict, confidence, scores, risk, reason, and repair instruction.

Why it matters: Machine-readable decisions make automation and auditing possible.

Approval states

Supports APPROVED, DISAPPROVED, and NEEDS_HUMAN_REVIEW.

Why it matters: Not every response is binary; risky cases can pause for a person.

Prompt repair loop

Rejected answers produce hidden corrective instructions and a retry request.

Why it matters: The app improves outputs instead of simply failing silently.

Automatic retry

Retries until approved or the endpoint retry limit is reached.

Why it matters: Gives upstream models a chance to correct mistakes before users see them.

Human review queue

Risky or uncertain outputs can be approved, rejected, edited, regenerated, exported, or annotated.

Why it matters: Critical workflows get human judgment when needed.

Audit logs

JSONL logs record decisions, retry chains, prompt repairs, and endpoint errors.

Why it matters: Post-incident review and hackathon demos have evidence.

Secret redaction

Authorization headers and API keys are hidden in TUI and logs.

Why it matters: Demos and screenshots remain safe to share.

Privacy mode

Can store metadata only instead of full prompts/responses.

Why it matters: Supports sensitive workflows and low-trust environments.

Per-endpoint profiles

Profiles tune criteria for coding, education, medical, finance, legal, research, disaster response, and multilingual use.

Why it matters: Different domains need different review strictness.

Model manager

Lists Gemma model choices and backends including Ollama, llama.cpp, LiteRT, Unsloth, and manual endpoints.

Why it matters: Users can choose speed, quality, or hardware fit.

Ollama support

Recommended local backend with configurable Gemma model tags such as gemma4:e2b and gemma4:e4b.

Why it matters: The easiest path for local Gemma judging.

llama.cpp support

Connects to local/resource-constrained GGUF inference servers.

Why it matters: Supports CPU/GPU constrained local deployments.

LiteRT edge prefilter

Experimental lightweight edge classification path before full judging.

Why it matters: Enables future low-latency edge safety checks.

Unsloth fine-tuning notebook

Public Colab notebook fine-tunes a Witness Gemma 4 E2B judge with LoRA/QLoRA.

Why it matters: Makes the judge specialization reproducible.

Hugging Face model support

The fine-tuned Witness Gemma 4 E2B judge adapter is published on Hugging Face.

Why it matters: Users can download and inspect the model artifact path.

Blackbox endpoint example

A ready endpoint template uses BLACKBOX_API_KEY through environment variables.

Why it matters: Shows real external endpoint use without committing secrets.

Doctor command

Checks backend, model, schema, proxy, logs, endpoint, and optional integrations.

Why it matters: Pre-demo verification is fast and transparent.

Curl installer

Install path can be scripted while still offering safer download-review-run steps.

Why it matters: Accessible for new users and judges.

Demo mode

Deterministic local judge and mock/demo endpoint make the workflow visible without external keys.

Why it matters: Hackathon judges can evaluate the idea even without private API credentials.

Structured verdicts

11. Gemma 4 Judge Schema

{
  "verdict": "APPROVED | DISAPPROVED | NEEDS_HUMAN_REVIEW",
  "confidence": 0.0,
  "safety_score": 0,
  "usefulness_score": 0,
  "prompt_alignment_score": 0,
  "correctness_risk": "low | medium | high",
  "rejection_reason": "",
  "suggested_fix": "",
  "improved_prompt_instruction": "",
  "requires_human_review": false
}
FieldPurpose
verdictAPPROVED, DISAPPROVED, or NEEDS_HUMAN_REVIEW.
confidenceHow confident the judge is in the decision.
safety_scoreSafety assessment for the candidate response.
usefulness_scoreWhether the answer is practically helpful.
prompt_alignment_scoreHow well the response follows the original user request.
correctness_riskLow, medium, or high risk of factual or operational incorrectness.
rejection_reasonHuman-readable reason for blocking the answer.
suggested_fixWhat must change before retrying.
improved_prompt_instructionHidden corrective instruction used for prompt repair.
requires_human_reviewWhether The Witness should pause for a person.
Repair loop

12. Prompt Repair

When Gemma rejects a response, The Witness creates a repaired request that preserves original user intent while adding hidden corrective instructions. It includes the rejection reason, the required fix, and stricter guidance after repeated failures.

Rejected attempt

Original request: Write a Python script that prints Hello World.

print(Hello World)

DISAPPROVED Python string is not quoted.

Repaired attempt

The previous answer was rejected because the Python string was not quoted. Generate a corrected answer using valid Python syntax.

print("Hello World")

APPROVED Returned to the app.

Internal details stay internal: prompt repair should not leak secrets or unnecessary judge internals to the final user.

13. Technology Tracks

Ollama

The default local judge backend runs Gemma 4 through Ollama. The TUI supports gemma4:e2b, gemma4:e4b, and custom Ollama models.

llama.cpp

The project includes a llama.cpp backend for local/resource-constrained inference using Gemma models.

LiteRT

The project includes a LiteRT edge prefilter path for lightweight classification before escalating to a full judge. This path should be treated as experimental unless fully configured.

Unsloth

The public Colab notebook uses Unsloth and the fine-tuned Witness Gemma 4 E2B judge is published as a Hugging Face adapter.

Cactus-ready

The architecture is Cactus-ready for a future mobile companion that routes lightweight checks on-device and heavier checks to the desktop Witness instance. Completed mobile support is not claimed.

Track badge wall: The Witness spans Ollama, llama.cpp, LiteRT, Unsloth, Hugging Face, and safety/trust positioning.

Security

16. Security and Privacy

  • API keys are never stored in docs.
  • Examples use environment variables.
  • Authorization headers are redacted.
  • Privacy mode can avoid storing full prompts/responses.
  • Logs can store metadata only.
  • The project avoids committing tokens or model weights.
Never paste real API keys into GitHub, screenshots, or videos. Use environment variables such as $BLACKBOX_API_KEY and keep local shell history clean when possible.

Secret redaction example

Authorization: Bearer ********
auth: bearer_env BLACKBOX_API_KEY
Known boundaries

17. Limitations

  • The Witness reduces risk but does not guarantee correctness.
  • Gemma judge decisions can be wrong.
  • High-risk medical/legal/financial outputs still need professionals.
  • Local performance depends on model and hardware.
  • LiteRT may be experimental if not fully tested.
  • Cactus/mobile companion is future work unless implemented.
  • Streaming support may be limited.
  • Fine-tuned model quality depends on training data.
Honest claim: The value of The Witness is a practical reliability layer, not a promise of perfect safety.
